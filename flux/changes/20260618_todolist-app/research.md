# Research: Todolist 桌面摆件应用技术选型研究

## 1. 技术栈选择

### Decision: Qt 6.5+ with QML and C++17

**Rationale:**
- Qt 是跨平台桌面应用开发的成熟框架，支持 Windows 和 macOS
- QML 声明式 UI 非常适合桌面摆件应用的动画效果和主题自定义需求
- Qt Quick 提供高性能的场景图渲染，支持 60fps 动画
- C++ 后端提供高性能的业务逻辑处理和数据访问
- Qt 生态完善，包含 Qt SQL（SQLite 支持）、Qt Charts（数据可视化）、Qt System Tray（系统托盘）等组件
- 静态编译可以生成独立可执行文件，便于分发

**Alternatives considered:**
- **Electron**: 生态成熟，开发效率高，但包体积大（~100MB+），内存占用高，不适合桌面摆件场景
- **Tauri**: 包体积小（~5MB），性能好，但生态较新，Qt 的 QML 动画能力更强
- **Flutter**: 跨平台 UI 优秀，但桌面端支持仍在发展中，C++ 集成不如 Qt 方便
- **原生开发 (WinUI/SwiftUI)**: 需要维护两套代码，开发成本高

## 2. 架构设计

### Decision: 三层分层架构（表现层 QML / 业务逻辑层 C++ / 数据访问层 C++）

**Rationale:**
- 关注点分离：UI 与业务逻辑严格分离，便于独立开发和测试
- 可测试性：业务逻辑层不依赖 UI，可以进行单元测试
- 可维护性：清晰的层次结构，便于后续功能扩展
- 性能：C++ 后端处理数据和业务逻辑，QML 专注于 UI 渲染
- 符合 Qt 最佳实践：使用 Context Properties 进行 QML/C++ 桥接

**Alternatives considered:**
- **MVVM 架构**: 更适合复杂数据绑定，但对于本项目来说三层架构已经足够
- **MVC 架构**: 控制器层在 QML 中不太自然，三层架构更简洁

---

## 3. 数据存储方案

### Decision: SQLite 本地数据库 + JSON 序列化（GitHub 备份）

**Rationale:**
- **SQLite**:
  - 轻量级嵌入式数据库，无需单独的服务进程
  - Qt SQL 模块提供完善的支持
  - 支持事务、索引、复杂查询，性能优秀
  - 单文件存储，便于备份和迁移
- **JSON 序列化**:
  - 人类可读的格式，便于 Git 版本控制
  - 与 GitHub 仓库备份完美契合
  - nlohmann/json 是成熟的 C++ JSON 库，API 友好
  - 可以清晰地看到每次同步的变更内容

**Alternatives considered:**
- **纯文件存储 (JSON/YAML)**: 查询性能差，不适合复杂的数据统计
- **PostgreSQL/MySQL**: 需要单独的服务进程，不适合桌面应用
- ** Realm**: 性能优秀，但商业授权限制，生态不如 SQLite 广泛

## 4. GitHub 同步方案

### Decision: libgit2 + GitHub REST API

**Rationale:**
- **libgit2**:
  - 纯 C 实现的 Git 库，可移植性好
  - 不需要依赖系统 Git 命令
  - 支持细粒度的 Git 操作控制
  - 可以进行冲突检测和解决
- **GitHub REST API**:
  - 用于验证 Token 权限和仓库信息
  - 可以获取仓库的基本信息和分支列表
  - 与 libgit2 配合使用，提供完整的 GitHub 集成体验

**Alternatives considered:**
- **调用系统 Git 命令**: 简单但不够灵活，依赖用户环境，错误处理困难
- **GitHub CLI (gh)**: 需要用户额外安装，不便于分发
- **仅使用 GitHub REST API**: 可以进行文件操作，但缺少完整的 Git 历史和冲突处理能力

## 5. 测试策略

### Decision: 测试金字塔（70% 单元测试 / 20% 集成测试 / 10% 端到端测试）

**Rationale:**
- **单元测试 (70%)**:
  - 测试业务逻辑层的各个 Service 类
  - 使用 Qt Test 框架，与 Qt 生态无缝集成
  - 测试数据验证、状态转换、错误处理等
  - 运行速度快，可以频繁执行
- **集成测试 (20%)**:
  - 测试服务层与数据访问层的交互
  - 测试完整的用户操作流程
  - 使用内存数据库进行测试，保证隔离性
- **端到端测试 (10%)**:
  - 测试完整的用户场景
  - 包括 GitHub 同步、主题切换、数据备份恢复等
  - 使用 Qt Quick Test 进行 UI 自动化测试

**Alternatives considered:**
- **Google Test**: 功能强大，但 Qt Test 与 Qt 集成更好
- **仅手动测试**: 无法保证回归质量，不适合长期维护

## 6. 代码质量标准

### Decision: 采用代码评审规范中的质量标准

**Rationale:**
- **功能正确性**:
  - 代码实现既定目标
  - 边界条件处理完善
  - 错误处理优雅
  - 现有功能不回退
  - 业务逻辑准确
- **代码质量**:
  - 自文档化的代码结构
  - 单一职责，适当的作用域
  - 无不必要的代码重复
  - 清晰一致的命名规范
  - 注释解释"为什么"而非"是什么"
- **架构合规性**:
  - 遵循项目架构模式
  - 适当的关注点分离
  - 无循环依赖
  - 合理使用接口/协议
  - 适当的耦合和内聚

**Alternatives considered:**
- **自定义编码规范**: 已有成熟的规范可供参考，无需重复造轮子

---

## 7. 项目结构

### Decision: 按分层和功能模块组织代码

**Rationale:**
- 清晰的目录结构，便于新成员理解
- 按功能模块组织，便于定位和修改代码
- 头文件和实现文件分离，符合 C++ 惯例
- QML 文件单独组织，便于 UI 设计师协作
- 测试代码与源代码分离，便于构建配置

**Alternatives considered:**
- **按类型组织（所有 .h 在一起，所有 .cpp 在一起）**: 不利于功能模块的定位
- **扁平结构**: 代码量增加后难以维护
